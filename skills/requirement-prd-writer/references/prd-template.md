# 需求说明模板

```md
# <需求名>需求说明

## 1. 需求概述

- 核心改动：<一句话说明本次最核心改动>。
- 生效结果：<一句话说明配置/功能生效后用户可感知的结果>。
- 默认口径：<未配置、默认值、存量兼容等关键默认规则；无则删除本行>。
- 原型文件：<本地评审版可写同目录原型文件；提交禅道版写“查看禅道附件”或删除本行>。
- 关联知识库：`<主功能点>需求知识库`、`<次要功能点>需求知识库`（如用户确认不需要知识库，写 `无，用户确认本次按独立新功能处理`）。

## 2. 功能改动

### 2.1 <功能点/页面/配置入口>

- 修改位置：<后台菜单、页面或操作入口>。
- 新增/调整内容：<配置项、页面字段、按钮、弹窗、状态或文案>。
- 展示与交互：<用户如何查看、搜索、选择、切换、跳转或操作>。
- 保存/生效规则：<保存哪些业务配置、何时生效、清空/默认如何处理>。
- 异常处理：<加载失败、无权限、目标不存在、不可访问等用户可感知异常>。

### 2.2 <可选：第二个功能点>

- 修改位置：
- 新增/调整内容：
- 展示与交互：
- 保存/生效规则：
- 异常处理：

## 3. 规则与边界

- 权限/优先级：<跨页面或跨角色的权限优先级；无则删除>。
- 兼容规则：<存量配置、旧数据、默认状态如何兼容；无则删除>。
- 一致性规则：<页面、明细、导出、入口之间需要保持一致的规则；无则删除>。
- 不处理范围：<本次明确不做的旧逻辑或模块>。
- 风险点：<用户体验、兼容、数据口径、权限边界等风险；无则删除>。

## 4. 页面截图

| 页面 | 截图 | 说明 |
| --- | --- | --- |
| <页面名称> | <本地评审版：`![页面名称](相对路径.png)`；提交禅道版：转为 HTML `<img>` 或写“查看禅道附件”> | <只说明截图对应页面和改动位置，不重复正文规则> |

## 5. 待确认问题

| 问题 | 影响 | 建议处理 |
| --- | --- | --- |
| 暂无 | - | - |

## 6. 版本记录

| 版本 | 日期 | 作者 | 变更说明 |
| --- | --- | --- | --- |
| v1.0 | <YYYY-MM-DD> | Codex | 创建需求说明。 |
```

## 使用说明

- 默认使用精简结构：`需求概述`、`功能改动`、`规则与边界`、`页面截图`、`待确认问题`、`版本记录`。
- `页面截图` 默认放在 `规则与边界` 后面；只有用户明确要求截图前置或后置到文末时才调整。
- 默认不要写 `改动范围`、`验收标准`、`数据与接口` / `接口与数据字段`；只有用户明确要求技术方案时才增加。
- 需求说明必须使用产品/业务语言，不写研发实现细节：不要出现接口名、数据库表/字段、代码变量、枚举名、类名、方法名、缓存方案、SQL、任务脚本、埋点 key 等技术实现内容。
- 可以写用户可感知或业务可配置的“配置项/页面字段/筛选项/状态/按钮/文案”，但不要写保存到哪个技术字段、接口怎么传、后端如何实现。
- `需求概述` 控制在 3-6 条，只写核心目标、默认口径、原型/知识库引用，不重复功能详情。
- `功能改动` 用产品/业务语言描述页面、配置、交互、保存和生效口径；不要拆前端/后端，不要把研发实现细节写进需求说明。
- `规则与边界` 只写跨页面规则、权限优先级、兼容、不处理范围、一致性和风险点，不重复 `功能改动` 中的入口、字段和交互步骤。
- 同一规则只落一个主位置：入口/字段/交互写在 `功能改动`；权限优先级/兼容/不处理范围写在 `规则与边界`；截图说明只描述截图对应页面和改动位置。
- 如确实需要补充研发口径，应单独建立“技术方案/实现说明”，不要混入需求说明正文。
- 正式 PRD 默认必须关联至少一个需求知识库；如果没有知识库，先与用户确认是否建立知识库。只有用户明确要求建立/更新知识库时，才使用 `$zentao-kb`、`$requirement-kb-creator` 或 `$requirement-kb-updater` 生成/更新正式知识库；否则只能说明知识库待建立或由用户确认本次不需要知识库。
- 如果用户确认不需要知识库，可以按独立新功能继续写正式 PRD；`需求概述` 的“关联知识库”写 `无，用户确认本次按独立新功能处理`。
- PRD 中建议保留必要表格，方便提交禅道时转成真实 HTML 表格；能用短列表表达的内容不要强行表格化。
- 版本记录默认只保留关键版本和当前版本，避免大量已被推翻的中间口径干扰阅读；完整过程可沉淀到知识库或禅道动态。
- 准备提交禅道的 PRD 不要包含本机路径或临时地址；页面截图可以随需求附件上传，并在提交时将 Markdown 图片转成 HTML `<img>` 标签引用禅道可访问的图片地址；如果无法获取可访问图片地址，则写“查看禅道附件 `<附件名>`”。
- 提交禅道时，Markdown 正文需转为 HTML：标题、列表、表格和图片都要是真实 HTML 结构；图片建议使用 `<img width="520" style="width:520px;height:auto;" />`，不要把 Markdown 图片语法或本机路径直接写入禅道正文。
- 本地最终回复可以列出绝对路径，方便用户打开；但禅道正文不能出现 `/Users/...`、`/home/...`、`C:\...`、`file://`、`localhost`、`127.0.0.1`。
