# requirement-kb-updater / 需求知识库更新

## 调用原则

只在用户目标是“更新已有需求知识库”时使用本 Skill。必须有用户明确写入意图，例如“更新知识库 / 同步知识库 / 补充到知识库 / 沉淀到知识库 / 记录到知识库”。如果用户只是要求写 PRD、设计需求、确认问题、整理方案或查询资料，不得调用本 Skill 写入正式知识库；只能输出待更新草稿或建议。

中文说明：本 Skill 的目标是把用户已确认要沉淀的新增或变更需求事实同步进已有 `需求知识库.md`，并保留版本变更记录；如果没有现有知识库，且用户明确要求建立知识库，才改用 `$requirement-kb-creator` 新建。

适用输入：

- 已有 `...需求知识库.md` 文件路径。
- 用户补充的新需求、新规则、纠错意见、原型调整、验收口径变化。
- 本地需求文件/文档：Markdown、Word、PDF、Excel、CSV、已有 PRD、会议纪要、测试反馈、截图、HTML 原型。
- 禅道任务/需求/缺陷的变更说明、备注、附件或 URL。
- 用户口令：`$requirement-kb-updater`、`知识库更新`、`更新需求知识库`、`同步知识库`、`基于本地文件更新知识库`、`从文档同步知识库`。

不适用输入：

- 新建知识库（可基于禅道、本地文件或用户描述）：使用 `$requirement-kb-creator`。
- 从知识库写 PRD：使用 `$requirement-prd-writer`。
- 生成 HTML 原型：使用 `$html-interactive-prototype`。


## 写入授权门禁

中文说明：正式知识库是已确认事实沉淀，不是 PRD 草稿缓存。除非用户明确要求更新知识库，否则不得修改 `...需求知识库.md`。

允许写入正式知识库的表达包括：

- `更新知识库`、`同步知识库`、`补充知识库`、`沉淀到知识库`、`记录到知识库`。
- `把这些规则写入/同步到 xxx需求知识库.md`。
- 用户在你询问“是否更新知识库”后明确回复“确认/可以/按这个更新”。

不允许写入正式知识库的场景：

- 用户只是说“帮我写需求/设计需求/生成 PRD/给方案/列待确认问题”。
- 用户只是在回答 PRD 待确认问题或调整方案，未明确说同步知识库。
- 用户只是要求查询资料、查禅道、看截图、整理口径。

未授权时的处理：

- 可以读取现有知识库和来源材料。
- 可以输出“拟更新项 / 本轮规则摘要 / 知识库待更新草稿 / 禅道检索摘要”。
- 不得覆盖正式知识库正文，不得新增版本记录，不得删除待确认问题，不得把 AI 推导内容写成确认规则。

## 推荐口令

- `$requirement-kb-updater 把这些新规则同步到 xxx需求知识库.md，并记录版本变更`
- `知识库更新：根据最新需求更新视频展示区的需求知识库`
- `更新需求知识库，补充版本变更记录，不要加入数据与接口章节`
- `基于这个本地 PRD 更新 xxx需求知识库.md`
- `从 ./最新需求文档/ 同步规则到现有知识库`

## 工作流程

1. **读取现有知识库**
   - 打开用户指定的 `...需求知识库.md`。
   - 识别现有章节、编号、版本记录、待确认问题。
   - 同时识别 `需求来源`、`关联任务/记录`、`禅道历史需求检索记录`、版本记录中是否已经保存过禅道检索摘要；记录已查产品/项目、关键词、命中 ID 和已读取详情 ID。
   - 如果用户没给路径，先在当前需求目录查找 `*知识库*.md`，只有多个候选时再询问。

2. **读取变更来源**
   - 本地 Markdown/文本/PRD：直接读取并抽取新增规则、变更规则、删除规则、验收口径和待确认点。
   - Word/PDF/Excel/CSV：使用可用工具提取文字、表格和图片；记录来源文件名、页码/表名等线索。
   - HTML 原型：读取页面文案、控件、交互脚本和状态分支，识别对知识库规则的影响。
   - 截图/图片：识别文字、表格、编号、箭头、UI 状态和字段说明；不确定内容进入待确认。
   - 禅道变更：读取描述、备注、附件、状态和关联记录；不要把账号密码、token、cookie 写入知识库。
   - 如果本次更新涉及新功能点、新模块、新渠道、新端范围，或用户指出“历史需求/禅道关联/之前漏查”，必须先按功能点关键词检查禅道历史需求；已有同关键词检索记录且覆盖当前产品/项目时可复用，不必重复全量查询。
   - 若复用已有禅道检索记录，要在本次版本记录或关联记录中写明“复用既有检索记录”；若新增检索，要补充检索日期、关键词、命中 ID、已读取详情 ID。中文说明：知识库更新时也要维护可复用检索记录，避免后续重复拉取禅道历史需求。
   - **禅道来源必须结合图片/附件处理**：如果禅道 `desc`、`spec`、`verify`、`customDemandSpec`、备注/动态中包含 `<img>`、文件链接、`fileID`、`{123.png}` 或原型链接，必须下载或打开附件并识别图片内容；不能只抽取文本后把图片里的页面字段、筛选项、按钮、红框/箭头说明写成待确认。
   - 从禅道 HTML 中提取 `img src`、`m=file&f=read`、`fileID=` 等链接，保存附件到需求目录或 `/tmp` 后识别；附件无法访问时，在版本记录、图片识别摘要或待确认问题中记录 story/task/bug ID、fileID、失败原因和影响范围。
   - 用户口述：作为“用户本轮补充”来源记录，缺少边界时写待确认。

3. **解析新变更**
   - 先再次确认本轮有用户明确的知识库写入授权；没有授权时停止在“拟更新项/草稿”输出，不进入正式写入步骤。
   - 把来源材料拆成可落地的知识库更新项。
   - 区分：新增规则、修改规则、删除规则、待确认问题、原型交互调整、验收口径调整。
   - 多来源冲突时，按“用户本轮明确口径 > 最新本地需求文件 > 禅道最新变更 > 历史知识库”排序；仍无法判断时进入 `待确认问题`，不要写成确定需求。

4. **更新对应章节（仅限已授权写入）**
   - `核心需求概述`：补充或改写最高层规则。
   - `后台/配置需求`：更新后台入口、字段、校验、联动、保存、删除等规则。
   - `客户端/前端展示需求`：更新 PC/移动端展示、跳转、隐藏、空态、异常态等规则。
   - `验收标准`：同步新增或修改可测试验收点。
   - `图片识别摘要`：只有新图片或图片解读变化时更新；若本次来源是禅道且包含图片/附件，必须补充或更新该摘要，列出 story/task/bug ID、fileID/附件名和识别内容。
   - `待确认问题`：更新仍不确定的问题，删除已被确认的问题。
   - `关联任务/记录`：如有禅道历史检索结果，补充或更新 `禅道历史需求检索记录` 表；已有相同关键词记录时合并命中 ID 和详情读取状态，不要重复堆多份相同记录。

5. **维护版本变更记录（仅限已授权写入）**
   - 如果知识库没有版本记录，新增 `## <n>. 版本变更记录` 作为最后一章。
   - 使用表格：`版本 | 日期 | 变更来源 | 变更说明`。
   - 新版本号按已有记录递增：`v1.0 -> v1.1 -> v1.2`；若无记录，从 `v1.0` 开始。
   - 日期使用当前会话日期。
   - 变更来源要写清楚来源类型，例如：`本地 PRD：xxx.md`、`PDF：xxx.pdf`、`禅道 taskID=xxx`、`用户本轮补充`。
   - 变更说明要概括本次真实变更，不写空话。

6. **清理与校验**
   - 保持章节编号连续。
   - 默认不要添加 `数据与接口` / `接口与数据字段` 章节，除非用户明确要求。
   - 检查新规则在知识库正文和验收标准中都有体现。
   - 检查本地来源文件/禅道来源/用户补充都在版本记录或关联记录中可追溯。
   - 如果本次新增或纠正功能点知识库来源，必须校验是否已处理禅道历史需求：已查命中、已查未命中、复用既有检索记录、或检索失败原因四选一。
   - 如果本次更新来源包含禅道任务/需求/缺陷，必须校验禅道图片/附件是否已提取、下载/打开、识别并同步到正文、验收标准或图片识别摘要；未处理附件时不能宣称知识库已完整同步。
   - 检查是否误留敏感信息、临时说明、本机临时路径或 AI 自述。

## 版本记录写法

参考 `references/version-log-template.md`。

版本说明示例：

```md
| v1.1 | 2026-04-25 | 用户补充的新需求 | 标题改为非必填；空标题隐藏 logo/名称且无占位；单游戏直跳隐藏游戏列表和二级分类入口；视频右上角增加声音开关。 |
```

## 输出约定

最终回复包含：

- 更新的知识库绝对路径；如果未获授权，只输出草稿/拟更新项路径或说明未写入正式知识库。
- 本次同步的关键规则摘要。
- 使用的变更来源：本地文件路径/禅道 ID/用户补充等。
- 新增的版本号。
- 校验结果：章节编号是否连续、是否无 `数据与接口` 章节、是否已同步验收标准。
